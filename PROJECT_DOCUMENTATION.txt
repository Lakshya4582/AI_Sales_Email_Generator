================================================================================
  1. WHAT THIS PROJECT IS
================================================================================

A web application that helps sales teams draft professional outreach emails
using an AI model. The tool focuses on the *sales workflow* — not just
generating emails, but all the related tasks a sales rep does daily:

  * Write a new cold email from scratch
  * Write a follow-up when a prospect doesn't reply
  * Brainstorm 5 subject line variations
  * Polish an existing draft
  * Save and reuse past emails
  * Edit results inline
  * Send via Gmail or any email client in one click

The entire app runs on your own computer by default using Ollama (a free,
local AI model runner), so there's no API cost. With a 3-line config change,
it can also use OpenAI, Google Gemini, Anthropic Claude, Groq, or any other
OpenAI-compatible LLM provider.

Built as an MVP — not yet deployed publicly, no billing, no multi-tenant
workspaces. Intended as a personal/demo product that can be extended toward
a real B2B SaaS.

================================================================================
  4. HOW THE WHOLE THING WORKS (END-TO-END FLOW)
================================================================================

STEP 1 — User opens signup.html in browser
  - Types first name, last name, email, password, company (optional), role.
  - Frontend sends POST /signup to backend.

STEP 2 — Backend receives signup
  - Pydantic validates input (email format, password ≥8 chars, etc.).
  - Checks if email already exists in users table → if yes, 409 error.
  - Hashes password with bcrypt (produces $2b$12$... hash).
  - Inserts new row into users table.
  - Generates a JWT signed with AUTH_SECRET, containing the user's email
    and expiry (7 days).
  - Returns {access_token, email} to the frontend.

STEP 3 — Frontend stores token
  - auth.js saves the token and email in localStorage.
  - Browser is redirected to index.html.

STEP 4 — index.html checks auth on load
  - auth.js's requireAuth() runs immediately.
  - If no token → redirect to login.html.
  - If token exists → continue loading the page.

STEP 5 — User fills in product/audience/tone/length → clicks Generate
  - Frontend calls apiFetch("/generate-email", POST).
  - apiFetch automatically adds "Authorization: Bearer <token>" header.
  - Backend's current_user dependency verifies the JWT, loads the user
    from DB, attaches it to the request.
  - Rate limiter checks: has this IP made >10 requests in the last minute?
    If yes → 429 Too Many Requests.
  - Route calls generate_email_content() in services.py.

STEP 6 — services.py builds a prompt and calls the LLM
  - Constructs a prompt string with product, audience, tone, length, and
    the user's name/company/role (so the email is signed correctly).
  - Calls client.chat.completions.create(model=MODEL, messages=...).
  - The OpenAI SDK sends this to whatever BASE_URL is in .env.
  - If .env points to Ollama → the request goes to localhost:11434/v1
    and Ollama generates the email locally.
  - The response text comes back.

STEP 7 — Route saves to DB and returns
  - A new row is inserted into email_history table with product, audience,
    tone, length, result (full generated text), and user_id.
  - The route returns {"result": "...", "history_id": 42} to frontend.

STEP 8 — Frontend renders the email
  - script.js's parseEmail() function splits the raw text into
    Subject / Body / CTA sections (handles markdown asterisks, numbered
    labels, etc.).
  - Each section is shown in its own styled block with Copy buttons.
  - History panel reloads to show the new entry.

STEP 9 — User edits / sends / saves
  - Each section is contenteditable, so user can click and type.
  - "Send to" + "Open in Gmail" builds a Gmail compose URL with the
    edited subject/body/to pre-filled.
  - "Save edits" calls PATCH /history/{id} to persist changes to DB.


================================================================================
  5. FEATURES — ALL OF THEM, IN DETAIL
================================================================================

5.1  SIGNUP / LOGIN
-------------------
  Users create an account with:
    - First name, last name (required)
    - Email + password (≥8 chars, required)
    - Company, role (optional)
  Passwords are hashed with bcrypt before storage.
  Login returns a JWT valid for 7 days.
  Token is stored in browser localStorage (key: "aiemail_token").

5.2  COLD EMAIL GENERATOR (the original feature)
------------------------------------------------
  Input: product, audience, tone (3 options), length (3 options).
  Output: a complete sales email with Subject, Body, CTA.
  Behavior: the email is signed off with the user's real name (from their
  profile), and includes their role/company when set.
  Endpoint: POST /generate-email
  Service: generate_email_content() in services.py
  Auto-saves to email_history.

5.3  FOLLOW-UP GENERATOR
------------------------
  Input: a past email from history, days since sent, optional "anything new"
  note.
  Output: a short, polite follow-up that:
    - Briefly references the original (doesn't repeat it)
    - Is noticeably shorter than the original
    - Has one clear, low-friction CTA
    - Incorporates the user's note if provided
  Stored with parent_id linking back to the original. Shows a "Follow-up"
  badge and indigo left-border in the history panel.
  Endpoint: POST /follow-up
  Service: generate_followup_content() in services.py

5.4  SUBJECT LINE BRAINSTORM
----------------------------
  Input: product, audience, tone (no length needed).
  Output: exactly 5 subject lines, each using a different angle:
    1. Direct benefit / value
    2. Curiosity / question hook
    3. Specific / data-driven
    4. Pain point / friction
    5. Low-friction / casual
  Under 60 characters each, no emojis, no ALL CAPS.
  A defensive parser handles LLM output variations (numbered lists, bullets,
  quotes, "Subject 1:" prefixes) and dedupes.
  Endpoint: POST /subject-lines
  Service: generate_subject_lines() + _parse_subject_lines()

5.5  EMAIL IMPROVER
-------------------
  Input: a rough draft (pasted textarea), optional target tone.
  Output: a polished rewrite with clear Subject, Body, CTA, signed with the
  user's real name.
  Keeps the user's intent and facts, but tightens language, strengthens
  verbs, cuts fluff.
  Endpoint: POST /improve-email
  Service: improve_email_content() in services.py

5.6  EMAIL HISTORY
------------------
  Every generated email (cold, follow-up, improved) is saved per user.
  History panel shows:
    - Product name
    - Audience / tone / length / date
    - Follow-up badge if it's a follow-up
    - View button (restores the form + result)
    - Follow up button (opens follow-up form for that entry)
    - Delete button (with confirm dialog)
  Endpoints: GET /history, DELETE /history/{id}, PATCH /history/{id}

5.7  INLINE EDITING
-------------------
  Every section of a generated email is contenteditable. User can click any
  section (Subject, Body, CTA) and type directly.
  Copy buttons, Send buttons, and Save Edits all read from the current
  (edited) DOM text.

5.8  ONE-CLICK SEND
-------------------
  A "Send to" field captures the recipient. Two buttons:
    - "Open in Gmail" — opens Gmail web compose in a new tab with
      subject/body/to all pre-filled via URL params.
    - "Default email" — opens whatever email client the OS has set as
      default (Outlook, Apple Mail, Thunderbird) via mailto: link.
  No OAuth, no infrastructure — just opens the user's existing email tool.

5.9  SAVE EDITS
---------------
  When viewing an email that came from history, the "Save edits" button
  appears. Clicking it PATCHes /history/{id} with the current edited text,
  persisting the changes.

5.10 PROFILE EDITING
--------------------
  profile.html lets the logged-in user:
    - Update first name, last name, company, role
    - Change password (requires current password for security)
  Email address cannot be changed.
  Endpoints: PATCH /me, POST /change-password

5.11 RATE LIMITING
------------------
  Per-IP limits applied to expensive/abuse-prone endpoints:
    - /signup:         5/minute,  30/hour   (anti-spam signup)
    - /login:         10/minute, 100/hour   (anti-brute-force)
    - /generate-email:10/minute, 100/hour
    - /follow-up:     10/minute, 100/hour
    - /improve-email: 10/minute, 100/hour
    - /subject-lines: 20/minute, 200/hour   (cheaper endpoint)
  Over-limit requests get 429 Too Many Requests.

================================================================================
  10. AUTHENTICATION SYSTEM
================================================================================

HOW IT WORKS:
  1. User submits /signup or /login with email + password.
  2. Backend verifies or creates the user.
  3. Backend generates a JWT:
       header:  {"alg": "HS256", "typ": "JWT"}
       payload: {"sub": "user@example.com", "exp": <unix-timestamp>}
       signed with AUTH_SECRET using HS256.
  4. Token returned in response body.
  5. Frontend stores in localStorage (key: "aiemail_token").
  6. On every subsequent request, auth.js attaches:
       Authorization: Bearer <token>
  7. Backend's current_user dependency:
       - Extracts the token from the Authorization header.
       - Decodes+verifies signature with AUTH_SECRET.
       - Checks exp is in the future.
       - Loads the user from the DB by email.
       - Returns the User object, which FastAPI injects into the route.
  8. If any step fails → 401 Unauthorized.
  9. On 401, auth.js's apiFetch automatically clears the token and
     redirects to login.html.

TOKEN LIFETIME: 7 days (TOKEN_TTL_HOURS = 24 * 7 in auth.py).

PASSWORD STORAGE:
  Never stored in plain text. Flow:
    hash = bcrypt.hashpw(password.encode(), bcrypt.gensalt())
  bcrypt automatically adds a salt and hashes at 12 cost factor.
  Stored string looks like: $2b$12$kjPc8kSWrFuwqwCGH6s09u5...

  On login verify:
    bcrypt.checkpw(password.encode(), stored_hash.encode())

SECURITY NOTES:
  - AUTH_SECRET should be at least 64 random bytes; set in .env.
  - If AUTH_SECRET changes, all existing tokens are invalidated.
  - Tokens are NOT revokable server-side (stateless JWT) — if you need
    logout-on-server, add a token blacklist table. Not yet implemented.
  - In production over HTTPS, localStorage is acceptable. Over plain HTTP,
    tokens can be intercepted — HTTPS is mandatory before deploying.


================================================================================
  11. LLM INTEGRATION — PROVIDER-AGNOSTIC DESIGN
================================================================================

KEY INSIGHT: most modern LLM providers expose an OpenAI-compatible endpoint.
This means the same code can talk to OpenAI, Ollama, Groq, Gemini, Claude,
DeepSeek, etc. The only thing that changes is three .env variables.

HOW IT'S IMPLEMENTED:
  services.py:
    client = OpenAI(api_key=API_KEY, base_url=BASE_URL)
    client.chat.completions.create(model=MODEL, messages=[...])

  The OpenAI Python SDK doesn't care that you're calling Ollama — it just
  sends an HTTP request to BASE_URL using the OpenAI API schema.

DEFAULT CONFIG (in .env):
  OPENAI_API_KEY=ollama                      (any string works for Ollama)
  OPENAI_BASE_URL=http://localhost:11434/v1  (Ollama's OpenAI-compat endpoint)
  OPENAI_MODEL=llama3.2:3b                   (whatever model you've pulled)

PROMPT DESIGN:
  Each service function builds a specific prompt that tells the model:
    - What it should produce (a sales email, a follow-up, subject lines, etc.)
    - Context (sender's name, company, role — for signing emails correctly)
    - Format (Subject:, Email:, CTA: markers so the frontend can parse)
    - Constraints (length limits, style, what not to do)

ERROR HANDLING:
  If the LLM backend is unreachable (Ollama off, wrong URL, etc.),
  APIConnectionError is raised. _call_llm catches it and returns a
  helpful message that tells the user what to do:
    "Could not reach model backend at <URL>. If using Ollama, make sure
     it is installed and running..."

PARSING THE OUTPUT:
  Different LLMs format their responses differently. llama3.2:3b often
  uses markdown (**Subject:**), while gpt-4o-mini uses plain text.
  script.js's parseEmail() is defensive:
    - Strips all ** markdown
    - Matches labels case-insensitively: Subject, Email, Body, Message,
      CTA, Call to Action
    - Supports both "Subject: foo" and "Subject:\nfoo" layouts
    - If no labels found, puts everything in the Body section (fallback)


================================================================================
  12. RATE LIMITING
================================================================================

LIBRARY: slowapi (a FastAPI-compatible port of Flask-Limiter).

KEY FUNCTION: get_remote_address — uses the requesting client IP.

HOW IT'S APPLIED:
  - main.py attaches the limiter to app.state and adds error handler +
    middleware.
  - routes.py decorates individual endpoints:
      @router.post("/generate-email")
      @limiter.limit("10/minute;100/hour")
      def generate_email(request: Request, ...):
  - The `request: Request` parameter is REQUIRED — slowapi uses it to
    extract the IP. FastAPI injects it automatically.

STORAGE:
  Limits are kept in-memory by default. This means:
    - Restarting the backend resets all counters.
    - Doesn't work across multiple backend processes (use Redis for that).
  For single-process dev/demo use, in-memory is perfect.

RATES:
  See section 5.11 above for the table of limits.

RESPONSE WHEN LIMIT HIT:
  HTTP 429 Too Many Requests
  Body: {"error": "Rate limit exceeded: <limit>"}

IMPORTANT LIMITATION:
  Key is per-IP. In a corporate office with NAT, all users share one IP.
  For real multi-tenant production: switch the key function to use the
  authenticated user email (requires a small refactor).


================================================================================
  13. SECURITY MEASURES
================================================================================

✅ IMPLEMENTED:
  - bcrypt password hashing (cost 12, salted per-password)
  - JWT with HS256 signature, 7-day expiry
  - Rate limiting on auth + LLM endpoints
  - Pydantic validation on every request body (blocks malformed input)
  - SQLAlchemy ORM = parameterized queries (no raw SQL concatenation =
    no SQL injection)
  - Ownership checks on /history and /follow-up (can't access others'
    emails even if you guess an id)
  - .env, .env.local, app.db, venv/ in .gitignore (no secrets leak on
    git push)
  - CORS configured with allow_credentials=False (safe with wildcard origins)
  - Generic login error message ("Incorrect email or password") — doesn't
    reveal whether the email exists
  - Change-password requires current password

❌ NOT YET IMPLEMENTED (for a real production deploy):
  - HTTPS (required before deploying — tokens over plain HTTP are stealable)
  - Password reset via email
  - Email verification on signup
  - Account lockout after N failed login attempts
  - Server-side token revocation / logout blacklist
  - CSRF protection (not needed for Bearer token auth, but needed if you
    switch to cookie auth)
  - Content Security Policy headers
  - Input sanitization for XSS — currently we use textContent everywhere
    in JS which is safe, but innerHTML anywhere would need escaping
  - Dependency vulnerability scanning (run `pip audit` before deploy)


================================================================================
  14. SETUP AND RUN INSTRUCTIONS
================================================================================

ONE-TIME SETUP:
  1. Install Python 3.12 from https://python.org (add to PATH).
  2. Open the project in a terminal:
       cd D:\ai-sales-email-generator
  3. Create a virtual environment:
       python -m venv venv
  4. Activate it:
       venv\Scripts\activate        (Windows)
  5. Install dependencies:
       pip install -r requirements.txt
  6. Install Ollama from https://ollama.com/download/windows
  7. Pull the model:
       ollama pull llama3.2:3b
  8. Create .env (copy from .env.example):
       copy .env.example .env
  9. Generate a random AUTH_SECRET:
       python -c "import secrets; print(secrets.token_urlsafe(64))"
     Paste the output into .env's AUTH_SECRET line.

EVERY TIME YOU RUN IT:
  1. Make sure Ollama is running (it's in the Windows system tray after
     install; if not, launch the Ollama app).
  2. Activate venv:
       venv\Scripts\activate
  3. Start backend:
       python -m uvicorn backend.main:app --host 127.0.0.1 --port 8000
  4. Open frontend/signup.html in a browser (any browser works).
  5. Create an account, log in, use the app.
  6. To stop: Ctrl+C in the terminal running uvicorn.

DURING DEVELOPMENT:
  - Add --reload to uvicorn to auto-restart on code changes:
       python -m uvicorn backend.main:app --reload
  - --reload only watches .py files. Changes to .env require a full
    restart (stop Ctrl+C, then re-run).

TROUBLESHOOTING:
  "Failed to fetch" in browser       → Backend not running. Start uvicorn.
  "401 Incorrect API key"            → Wrong/placeholder API key in .env.
  "Could not reach model backend"    → Ollama isn't running. Launch Ollama app.
  "429 Too Many Requests"            → Rate limit — wait a minute.
  "ModuleNotFoundError"              → Forgot to activate venv, or missed
                                       `pip install -r requirements.txt`.
  DB schema mismatch after model change → startup migration covers
                                       adding columns; if bigger changes
                                       needed, delete app.db and restart.

================================================================================
  17. KNOWN LIMITATIONS
================================================================================

PRODUCT:
  - No password reset email (users who forget password are locked out).
  - No email verification on signup.
  - No multi-user workspaces / teams.
  - No billing / subscription.
  - Email history limit = 100 (configurable in GET /history limit param).
  - No bulk mode (can't upload a CSV of leads).
  - No templates / saved presets.
  - No CRM integrations (HubSpot, Salesforce).
  - No analytics (open rates, reply rates).

TECHNICAL:
  - SQLite is single-writer — won't scale past ~1000 users. Swap to
    Postgres by changing DATABASE_URL when needed.
  - In-memory rate limiting — doesn't work across multiple uvicorn workers.
    Use Redis-backed slowapi for prod.
  - Small local models (llama3.2:3b) sometimes ignore prompt instructions
    (e.g. leave `[Your Name]` placeholders). Switch to a bigger model
    via .env when needed.
  - First LLM call after server start is slow (5–15s model load into RAM).
    Subsequent calls are faster.
  - Follow-up prompt gives the model the FULL original email, which
    consumes tokens — fine for Ollama (free), expensive at scale with
    paid APIs. Could switch to summary-based follow-ups for cost.
  - No streaming — user waits for full response. Would improve UX a lot.

SECURITY:
  - No HTTPS yet (required before any public deploy).
  - No Content-Security-Policy header.
  - Tokens in localStorage — fine on HTTPS, vulnerable on plain HTTP.

================================================================================
  19. DEPLOYMENT GUIDE — WHAT WORKS AND WHAT CHANGES
================================================================================

THE BIG PICTURE
---------------
Once deployed, the app works on ANY device with a browser — phones, tablets,
laptops, other people's PCs. You just share a URL. The UI is already
responsive (has a mobile breakpoint at 520px).

Currently, everything runs on localhost:
  Your browser -> Your laptop's backend (localhost:8000)
               -> Your laptop's Ollama (localhost:11434)

After deployment, it becomes:
  Any browser anywhere -> Cloud backend (e.g. api-myapp.onrender.com)
                       -> Cloud LLM provider (e.g. Gemini API)

THE BIG GOTCHA: OLLAMA DOES NOT TRAVEL
--------------------------------------
The cloud server you deploy to will NOT have Ollama installed. It cannot
reach your laptop's Ollama either (different network, firewalled).

Before deploying, you must pick one:

  OPTION A — Switch to a cloud LLM (recommended)
    3-line change in .env. No code changes.
    See section 16 for exact values for each provider.
    Cost: $0 on Gemini free tier, ~$0.001/email on OpenAI.

  OPTION B — Deploy Ollama too (expensive, overkill)
    Requires a server with 8+ GB RAM and ideally a GPU.
    Costs $30-$200/month. Only worth it if data must never leave
    your infrastructure.

WHAT HAS TO CHANGE BEFORE DEPLOYING
-----------------------------------
1) Frontend API URL
   Currently hardcoded in frontend/auth.js and frontend/script.js:
     const API_URL = "http://127.0.0.1:8000/generate-email";
     const API_BASE = "http://127.0.0.1:8000";
   Change these to the real backend URL (or make it configurable via a
   small config.js that gets swapped per environment).

2) CORS restriction
   In backend/main.py:
     allow_origins=["*"]      <-- too permissive for production
   Change to:
     allow_origins=["https://myapp.vercel.app"]

3) HTTPS
   Required (not optional). Tokens in localStorage are safe over HTTPS;
   over plain HTTP they can be intercepted. All major hosts (Vercel,
   Render, Railway, Fly) give free HTTPS automatically via Let's Encrypt.

4) Database persistence
   On Render free tier, the disk is ephemeral — SQLite's app.db file
   will be lost on redeploy. Two options:
     a) Attach a persistent disk (Render paid, or Fly volumes)
     b) Switch to Postgres by changing DATABASE_URL in .env:
        DATABASE_URL=postgresql://user:pass@host:5432/dbname
     No code change needed — SQLAlchemy handles both.

5) LLM provider
   Switch from Ollama to a cloud provider (section 16 has exact .env
   values for Gemini / OpenAI / Groq / Claude).

WHERE TO DEPLOY
---------------
TYPICAL SPLIT:
  FRONTEND (static HTML/CSS/JS)
    Vercel       — fastest CDN, simple, free forever.
    Netlify      — similar, equally good.

  BACKEND (FastAPI)
    Render       — free tier, sleeps after 15 min idle.
    Fly.io       — free tier (3 small VMs, 3 GB storage).
    Railway      — was free, now $5/month minimum.
    AWS / GCP    — only if you know what you're doing (complex).

  DATABASE (optional, if switching off SQLite)
    Supabase     — Postgres + auth (can replace our auth too if wanted).
    Neon         — serverless Postgres, cold-starts in <1s.

STEP-BY-STEP DEPLOYMENT (high level):
  1. Push code to a GitHub repo (make sure .env is gitignored).
  2. Create Vercel account, connect the repo, point to frontend/ folder.
     -> Frontend is live at yourapp.vercel.app.
  3. Create Render account, connect the repo, configure:
       Build command:  pip install -r requirements.txt
       Start command:  uvicorn backend.main:app --host 0.0.0.0 --port $PORT
     Add environment variables: OPENAI_API_KEY, OPENAI_BASE_URL,
       OPENAI_MODEL, AUTH_SECRET, DATABASE_URL.
     -> Backend is live at yourapp.onrender.com.
  4. Update frontend API URL to point to the Render backend URL, push.
     Vercel auto-redeploys.
  5. Tighten backend/main.py CORS to the Vercel domain, push.
     Render auto-redeploys.

TOTAL TIME: ~2-3 hours once you know the steps.
TOTAL COST: $0 for a demo. $5-20/month at small real-user scale.
